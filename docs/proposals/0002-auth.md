## 1. Goal & scope

Minimum accounts to (a) gate premium features later (billing, Step 5) and (b) make watchlists/portfolios account-bound instead of browser-local. **Phase-1 minimum, nothing more.**

**In scope:** email/password sign-up + login, sessions, logout, password reset, account deletion, and migrating the browser-local watchlist into the account on first login.
**Out of scope (later/never):** roles/admin, teams, SSO beyond optional Google, anything touching funds (scope lock).

---

## 2. Guardrails that shape this (G4 / NDPA-NDPC)

- **Minimise PII:** store only email + auth essentials. No phone, address, BVN, etc.
- **Never log PII or secrets** (emails, tokens, password hashes) — add a redaction rule to the logger and keep auth state out of the AI layer.
- **Lawful basis + rights:** document basis before collecting any field; support **account deletion** (erasure) and data export from day one.
- **Security:** Argon2id/bcrypt password hashing, httpOnly+SameSite cookies, CSRF protection on mutations, rate-limit login/reset, generic error messages (no account enumeration).
- **Disclaimers/scope unchanged:** accounts don't change the "general information, not advice" posture (G2).

## 3. Schema additions (new migration — human-run)

Adds to the Prisma schema from 0001 (auth tables only):

| Table | Key columns | Notes |
|---|---|---|
| `users` | `id` UUID PK, `email` citext unique, `password_hash`, `email_verified_at?`, `created_at`, `tier` (`free`/`premium`, default free) | `password_hash` = Argon2id. Never logged. |
| `sessions` | `id`, `user_id` FK, `expires_at`, `created_at` | Only if we use DB sessions (D2). |
| `password_reset_tokens` | `id`, `user_id` FK, `token_hash`, `expires_at`, `used_at?` | Store a hash of the token, not the token. |
| `watchlist_items` | `user_id` FK, `ticker`, `added_at`, PK `(user_id, ticker)` | Account-bound watchlist; supersedes localStorage. |

`tier` lives on `users` now so billing (Step 5) just flips it. No portfolio tables yet (that's Step 4).

## 4. Approach options (decisions)

### D-A — Auth library vs hand-rolled
- **Auth.js (NextAuth v5)** — batteries-included for Next App Router; Credentials + OAuth providers; Prisma adapter. Most common, well-audited. *Recommended.*
- **Lucia** — lightweight, explicit, you own the flows; more code, fewer magic edges.
- **Hand-rolled** — most control, most footguns (not recommended for money-adjacent auth).

### D-B — Sign-in methods
- **Email/password only** (simplest Phase-1 minimum), or
- **Email/password + Google OAuth** (diaspora users likely expect Google; the old site had Google fields). *Recommend email/password first, Google as a fast-follow toggle.*

### D-C — Session strategy
- **DB sessions** (revocable, simple mental model, needs the `sessions` table), or
- **JWT cookie sessions** (stateless, no table, harder to revoke). *Recommend DB sessions for revocability.*

### D-D — Watchlist migration
On first successful login, merge any browser-local watchlist into `watchlist_items` (union), then switch the UI to read/write the account list. Keep localStorage as the anonymous fallback for logged-out users. *(Recommended — no data loss for existing anonymous users.)*
